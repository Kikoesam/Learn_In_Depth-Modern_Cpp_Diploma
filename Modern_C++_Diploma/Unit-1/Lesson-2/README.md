> [!NOTE]
> ### üß† Understanding `inline` Functions in C++
>
> #### üîπ What is `inline`?
> - The `inline` keyword is a **compiler hint** suggesting that the function's **code should be expanded in-place** where it‚Äôs called ‚Äî instead of performing a regular function call.
> - It's **not a command**, just a **suggestion**. The compiler can ignore it.
>
> #### üöÄ Why Use `inline`? (Optimization Goal)
> - To **reduce function call overhead**, especially in small, frequently-called functions.
> - Skips the stack frame setup, jump, and return process of normal calls.
>
> #### üß† What Is Function Call Overhead?
> - Every function call costs:
>   - Saving registers
>   - Jumping to a new address
>   - Allocating a call stack frame
> - For **tiny functions** (like `getX()`), this overhead is larger than the logic itself!
>
> #### üõ† Relation to Intermediate Representation (IR)
> - In **IR**, inlining replaces a function call with the actual IR instructions.
> - This allows the optimizer to perform **constant folding**, **dead code elimination**, and more aggressive **loop unrolling**.
> - Example:
>   ```cpp
>   inline int add(int a, int b) { return a + b; }
>   int x = add(1, 2); // IR becomes: int x = 3;
>   ```
>
> #### üîÑ Inline & Function Overloading
> - `inline` works with **overloaded functions**.
> - The compiler still resolves the **best match** using matching rules.
> - It can choose to inline one overload and not another depending on size or complexity.
>
>
>> [!NOTE]
>> ### üß† What Does `inline` Really Do?
>> 
>> #### üß© Replaces Function Call with Code
>> - Inlining means the **compiler copies the function‚Äôs logic directly** into the place it's called.
>> - No function call, no return ‚Äî just the actual code pasted inline.
>> 
>> ```cpp
>> inline int add(int a, int b) { return a + b; }
>> int result = add(2, 3); // becomes: int result = 2 + 3;
>> ```
>> 
>> ---
>> 
>> #### üöÄ Why Use `inline`? (Optimization)
>> - Skips function call overhead:
>>   - No register save
>>   - No stack frame
>>   - No jump and return
>> - Speeds up small, repeated functions ‚Äî e.g., math helpers, getters.
>> 
>> ---
>> 
>> #### üõ† How It Helps Compiler Optimizations
>> - In the **IR stage**, the compiler replaces function calls with instructions directly.
>> - Allows deeper optimizations:
>   - ‚úÖ Constant folding: `add(2, 3)` ‚Üí `5`
>   - ‚úÖ Dead code removal
>   - ‚úÖ Loop unrolling
> 
> ---
> 
> ### üí° When to Use It?
> - Use `inline` for:
>   - Tiny functions (1‚Äì3 lines)
>   - Getters/setters
>   - Utility functions in headers
> - Don‚Äôt use it for:
>   - Big functions (code bloat)
>   - Functions used only once (pointless)

>
>
> ---
>
> ### üß™ Inline vs Debug/Release Modes
>
> | Mode       | Optimization | Inlining Behavior       | Goal                        |
> |------------|--------------|--------------------------|-----------------------------|
> | `Debug`    | ‚ùå No         | üîÅ Functions are **not** inlined | Easier debugging, readable stack traces |
> | `Release`  | ‚úÖ Yes        | ‚úÖ **Auto-inlining** enabled (even without `inline`) | Faster performance, smaller call overhead |
>
> - üîß **In Debug Mode**, inlining is avoided so you can **step through** your function calls.
> - ‚ö° **In Release Mode**, the compiler may inline aggressively, even for functions **not marked as `inline`**.
>
> ---
>
> ### üìù Summary:
> - Use `inline` for **very small, performance-critical functions**
> - Let the compiler decide ‚Äî it‚Äôs smarter in Release mode
> - Inlining boosts optimization but hides stack traces in debugging
> - Debug builds skip inlining for clarity, Release builds inline for speed
